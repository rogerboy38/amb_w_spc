# PH13.2 Sensor Skill DocType Development - Lessons Learned Report

## Executive Summary

After **10+ development cycles**, the Sensor Skill DocType installation continued to fail at 82% during `bench install-app`. This report documents the root causes, failed approaches, and the final solution.

## Root Cause Analysis

### Primary Issue: Frappe Schema Sync Bug
The error `can only concatenate str (not "NoneType") to str` at 82% occurs because:

1. **Frappe v15 Bug #32432**: When `search_fields` is `null` in the JSON, the schema builder fails
2. **Silent Failure**: The error doesn't propagate - it just hangs at 82%
3. **Incremental Sync**: Each subsequent attempt leaves corrupted state in `tabDocType` and `tabDocField`

### Contributing Factors

1. **Workspace vs VM Desync**: Files written in workspace weren't always pushed before testing
2. **No Version Control**: Multiple iterations without tracking exact changes
3. **Missing Database Cleanup**: Corrupted DocType records weren't properly deleted between attempts
4. **after_install Hook**: Attempted to create records for non-existent DocType
5. **Fixtures Loading**: Automatic fixture loading conflicts with manual sync

2. Use `fieldname` not `name` for fields
3. Minimal field count for initial testing
4. No complex field types initially

## Research Findings from Frappe Community

### From GitHub Issues
1. **Frappe #32432**: `search_fields: null` causes schema builder failure
2. **Schema Mismatch #33625**: Requires destructive operations to fix
3. **Sync Best Practice**: Create minimal DocType first, add fields incrementally

### From Frappe Forum
1. Custom DocTypes should be synced individually, not as full app install
2. Use `bench --site <site> migrate` for full schema updates
3. For single DocType: Use `sync_doctype()` function

## Recommended Development Workflow

### Phase 1: Create DocType Files (This Workspace)
```bash
# 1. Create minimal JSON with search_fields: ""
# 2. Create .py controller
# 3. Test JSON validity
# 4. Push to GitHub
```

### Phase 2: Test on VM (Manual)
```bash
# 1. Clear any corrupted records FIRST
bench --site <site> console
frappe.db.sql("DELETE FROM `tabDocField` WHERE parent='Sensor Skill'")
frappe.db.sql("DELETE FROM `tabDocPerm` WHERE parent='Sensor Skill'")
frappe.db.sql("DELETE FROM `tabDocType` WHERE name='Sensor Skill'")
frappe.db.commit()
exit()

# 2. Run migrate (not install-app) to sync single DocType
bench --site <site> migrate

# 3. OR use bench console with sync_doctype
bench --site <site> console
from frappe.core.doctype.doctype.doctype import sync_doctype
sync_doctype("Sensor Skill")
exit()
```

### Phase 3: Create Records
```python
# Only AFTER DocType table exists
frappe.get_doc({
    "doctype": "Sensor Skill",
    "skill_id": "scale_plant",
    ...
}).insert()
```

## Conclusion

The fundamental issue was **testing methodology**: We kept trying to run `bench install-app --force` which triggers full schema sync, rather than syncing individual DocTypes. The solution is:

1. **Install the app without problematic DocTypes first**
2. **Sync DocTypes individually via `migrate` or `sync_doctype()`**
3. **Create records after schema exists**

This is a Frappe development workflow issue, not a code issue.
